## Migration Plan: Forge 1.20.1 → 1.21.1

This is a significant jump (skips 1.20.2/1.20.4/1.20.6, which is where Forge made its biggest breaking changes — registries and networking were both overhauled). Here's the breakdown by area, roughly in dependency order:

### Phase 0 — Build/toolchain files
- `gradle.properties`: bump `minecraft_version=1.21.1`, `minecraft_version_range=[1.21.1,1.22)`, `forge_version` → latest 1.21.1 Forge build (currently in the 52.x line), `mapping_version` → a Parchment release for 1.21.1 (or drop to `official`/`stable` if Parchment lags)
- `mods.toml` (both copies — one looks stale/duplicated, worth consolidating): bump `minecraft` dependency `versionRange="[1.21.1,)"`, `forge` versionRange to match new major
- Java toolchain: 1.21.1 still targets Java 21 (not 17) — update `java.toolchain.languageVersion`

### Phase 1 — Core mod class & registries
- `SunBlockCore.java`, `ModBlocks.java`, `ModItems.java`, `ModSounds.java`, `ModLootModifiers.java`: `RegistryObject<T>` → `DeferredHolder<T, ?>`, `DeferredRegister.create(...)` API tweaks, `RegisterEvent`/`FMLCommonSetupEvent` signature checks. CreativeModeTab registration syntax changed (now uses `BuildCreativeModeTabContentsEvent` plumbing differently).

### Phase 2 — Networking rewrite (biggest change — affects `ModPackets`, `ServerDataS2CPacket`)
`SimpleChannel`/`NetworkRegistry.ChannelBuilder` is **gone** in 1.21. Must rewrite to the `CustomPacketPayload` + `PayloadRegistrar`/`IPayloadRegistrar` system registered via `RegisterPayloadHandlersEvent`:
- `ServerDataS2CPacket` becomes a `record` implementing `CustomPacketPayload`, with a `Type<T>` and a `StreamCodec` instead of manual `FriendlyByteBuf` read/write
- `ModPackets.register()` rewritten entirely around `event.registrar(MODID).playToClient(TYPE, CODEC, HANDLER)`
- `sendToPlayer`/`sendToClients` switch to `PacketDistributor` static helpers (API also changed — no more `INSTANCE.send(...)`)

### Phase 3 — Client-side (`HUD`, `ClientSetup`, `ClientForgeHandler`/`ClientEventHandler`/`ClientModHandler`, `KeyBindings`)
- HUD overlay rendering: `IGuiOverlay`/`registerGuiOverlays` replaced by the new `VanillaGuiLayers`/`LayeredDraw` gui-layer system in 1.21
- `GuiGraphics` API churn (text rendering, blit signatures)
- `ItemProperties.register` predicate/property function changes in `ClientSetup`

### Phase 4 — Items/Blocks/Datagen
- `ModBlockStateProvider`, `ModItemModelProvider`, `ModGlobalLootModifiersProvider`, `DataGenerators`: `PackOutput`/`CompletableFuture<HolderLookup.Provider>` constructor signature changes (already partially present in 1.20.1 but tightened in 1.21)
- `ResourceLocation` constructor is fully removed in favor of `ResourceLocation.fromNamespaceAndPath(...)` / `withDefaultNamespace`
- Tool/weapon item classes (`SolarPickaxe`, `SolarAxe`, etc.) — `Tiers`/`ToolMaterial` and item property constructor changes

### Phase 5 — Config & misc
- `ForgeConfigSpec` likely stays (Forge-specific, unlike NeoForge's `ModConfigSpec`) but verify `isLoaded()`/`ConfigValue` API parity
- `PlayerUtils`/`RenderUtils`/loot modifier classes — spot-check for renamed vanilla classes (loot table changes, `LootItemFunction` registration via `MapCodec`)

### Phase 6 — Compile & smoke-test pass
- Iterative `./gradlew compileJava` → fix errors file by file, then `runClient`/`runServer` smoke test, focusing on: socket connection still works, HUD renders, packets sync, items/blocks register and show in creative tab, configs load without the "correcting" loop returning.

---

**Heads up on scope**: Phase 2 (networking) and Phase 3 (HUD overlays) are the highest-risk, most labor-intensive pieces — both APIs were redesigned, not just renamed, so those files get substantially rewritten rather than patched. I'd plan to tackle those first since everything else is comparatively mechanical (renames/signature tweaks).
